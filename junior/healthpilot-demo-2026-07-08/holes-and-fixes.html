HealthPilot — Holes, Fixes & Honest Risks

Internal memo · Junior · 8 July 2026 · Confidential — never publish

I ran a deep review of the product and the Ted's integration, then fixed what was safe to fix in our sandbox tonight. Below is the honest picture: what I improved, what still needs Felix, and the things we should not overstate to CJ.

1. What I fixed tonight (in our sandbox, verified live)

  • Harborview now has its own logo. The demo clinic was falling back to the Ted's Health logo on the patient's first screen, quietly undercutting the white-label story. Generated a clean Harborview wordmark and seeded it. The white-label promise is now visible at screen one.
  • Clinician stops are now shown, not described. Seeded a scheduled consultation so the clinician's pre-brief workflow renders live, and stamped quiz content versions on results so the progress view shows its cross-version caveat. These were previously narration.
  • Repeatable one-command demo bring-up (demo-up.sh). Stands up fresh tunnels and boots the whole stack clean, then restores the demo clinic to its intended state. Run it shortly before the call to get a live link. The old link was three days stale and dead.

2. Holes that still need Felix (read-only repos — I can't touch these)

  • HIGHThe AI rate limiters are dead code. The limiters meant to cap the AI endpoints exist but are wired to nothing, so the paid AI endpoints are effectively uncapped per IP. One-line fix each. Left unfixed, a bad actor can run up a real OpenAI bill.
  • HIGHLab-result webhooks are unauthenticated. The endpoint that writes blood-test results into a patient record accepts any/empty signature ("skip for demo"). Anyone who guesses an ID could write fake results to a real patient. Not demo-visible; a genuine hole before real patients.
  • HIGHClinic API keys fall back to a publicly-known encryption key if an env var is unset. Fine locally; must be set on the server before any real clinic key is stored.
  • MEDAnonymous-intake claim can hijack tokenless intakes on legacy paths. Fresh intakes are safe.

I've packaged these for Felix. None affect Thursday's demo; all matter before a real clinic goes live.

3. Things we must NOT overstate to CJ

  • It runs on GPT-4o today, not Claude. Some internal docs imply Claude plus prompt caching plus versioned prompts. That isn't wired up. The engine is provider-swappable, so the honest line is "GPT-4o today, provider-agnostic by design."
  • The red-flag safety net is a keyword backstop, not intelligent triage. It catches emergency phrases in English and the escalation wiring around it is solid, but it's regex: evadable by paraphrase, and English-only while we pitch Spanish/German clinics. Sell it as "a safety backstop with clinician escalation," never "AI that diagnoses." Pushing it toward "it can tell the patient they have low testosterone" tips it into medical-device (MHRA) territory — a hard line.
  • Recommendations can look similar for similar patients. The catalogue has real depth, but the quiz captures one goal and few structured symptoms, so two same-goal patients who skip bloods get near-identical output. For the demo, use differentiated personas.
  • The conversion economics are instrumented, not proven. The attribution is wired, but there is zero real attributed revenue yet. The model rests on an assumed 8% blood-panel conversion. Frame as "instrumented, awaiting design-partner data."

4. Demo-prep checklist

  • Run demo-up.sh shortly before the call — the tunnel link changes each run and goes stale within a day or two.
  • Do not run the e2e test between reseed and demo (it mutates two demo patients). The bring-up handles the reseed.
  • Keep the public/patient window off any /platform/* path — it clears the clinic branding.
  • Deferred cosmetics: owner-console "median completion time" reads low on same-instant seeded data; the six-step signup claim funnel is worth shortening later.

5. Two things I did NOT do tonight, on purpose

  • Did not remove the production Firebase alias from the sandbox tree, though it's a real footgun (a stray deploy could hit real Ted's prod). It's a deploy-config change — Felix should confirm and make it.
  • Did not seed anything that could break a working demo. Every change was verified against the live schema; the consultation seed is guarded so it can't take the rest of the demo down with it.
Bottom line: the demo is stronger than it was this morning and every claim in it is one I can stand behind. The remaining holes are Felix's to close before a real clinic, and the talk-track honesty notes are the difference between a credible pitch and one that unravels under a sharp question.
