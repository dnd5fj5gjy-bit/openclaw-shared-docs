Data flow — how a Telegram message becomes an answer

  • User sends a Telegram message to an agent's bot (e.g., Felix → @QuartermasterBot).
  • Listener daemon (telegram_listener.py) long-polls the Telegram API and receives the message.
  • Listener checks allowed_users, then calls tmux send-keys to the agent's pane formatted as [Telegram message from <name> (chat_id: ...)]: <text> followed by Enter.
  • Claude Code (the agent) sees the prompt and processes it. Reads CLAUDE.md, per-user state, memory. May spawn Task sub-agents for non-trivial work.
  • Agent calls send_telegram.py --chat-id <id> --text "..." to reply. Telegram API delivers to user's phone.
  • Session is logged to workspace/sessions/YYYY-MM-DD.jsonl.

Resilience layers

🛡️ process-watchdog

Restarts hung Claude Code instances. Detects dead pane process or stale pane content for 30+ min.

📡 listener-watchdog

Restarts Telegram listeners that crash. Original watchdog from before May 5.

🌐 tunnel-watchdog

Observes the email-API cloudflared tunnel. Notifies Felix only on sustained failure (~15+ min).

💾 backup

Hourly encrypted snapshot to private GitHub repo. age-encrypted credentials.

🗒️ log-rotate

Hourly. Truncates any log over 50MB to its last 2000 lines, in-place (preserves writers' open FDs).

🔁 weekly-restart

Sundays 23:00. Clean restart of all agents to refresh long-lived contexts.

📊 health-summary

08:00 daily. Telegrams Felix a fleet-status digest (who's up, last backup, recent restarts, disk).
