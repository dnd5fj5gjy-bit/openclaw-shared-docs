01 · The problem

People run out of hours long before they run out of ideas.

Every business has employees who could do more - chase more leads, write more content, answer more customers, catch more admin - if there were simply more of them in the day. The usual answer is to hire, which is slow, expensive, and doesn't scale with a person's judgement, only their headcount.

What if instead of hiring more people, a business gave its existing people an agent - or a small team of agents - built specifically for their role? Not a generic chatbot bolted onto the business. A named, engineered agent that knows the job, does the repeatable parts of it, and reports back like a colleague would.

Time is the most valuable currency a business has. This gives a business and its team more of it - and lets them spend it on the work that actually needs their judgement, not the work that just needed their hours.

06 · How it's sold

Priced per agent. Simple enough to buy on a call.

The packaging mirrors headcount, which is a pricing model every business owner already understands intuitively - it's just cheaper, faster, and doesn't need holiday cover.

What "Build + monthly" means: Build is a one-off fee to engineer and deploy the agent(s) - the equivalent of onboarding a new hire. Monthly is the ongoing retainer to keep them running, monitored, and improved - the equivalent of a salary. Team Agents prices this per agent as more are added; Full Workforce drops the fixed Build/Monthly split entirely and prices bespoke against the headcount it's replacing. Figures aren't locked yet - final numbers follow once the build process is templated - but the hosting question is settled: running an agent costs a small slice of a rented cloud server, not a chunk of dedicated hardware, so the retainer is priced against oversight and delivery, not infrastructure risk.

Why that matters commercially: because each agent's running cost is negligible and scales in small, linear steps, margin holds even as the client book grows - there's no point at which onboarding the next client requires a capital purchase that eats into the retainer. That's true at the Single Agent tier and it stays true at Full Workforce.

The demo is the business, not a slide.

Every prospect conversation can open with "here's our own agent team, running our own business, right now" - a live reference no competitor pitching AI can match on a slide deck alone. That credibility compounds: the more roles Bear Witness engineers agents for internally, the deeper the proof gets before a single external client is signed.

This gives Olly a direct acquisition motion: go to any business owner, ask what role they'd hire for if cost and hours weren't the constraint, and show them it already exists inside Bear Witness.

It also fits how few of us there are: the three of us don't need to grow headcount to grow this. The agents are the delivery mechanism, not extra staff - engineering one more named agent for a new client costs time, not a hire. That's what makes single-agent and small-team packages realistic to sell and service starting now, without waiting for the business to be bigger first.

The same is true of the infrastructure underneath it: an agent is software making calls to Anthropic's API, not a physical box sitting under someone's desk. Every new client's agents run in their own isolated workspace on a rented server, alongside other clients' agents on the same machine - the exact multi-tenant pattern Ted, our own agent, already runs serving multiple people from one machine. Adding the next client is a software step, not a hardware order, so the number of clients we can serve isn't capped by how many machines BGV owns.

We can show, not just tell: for any call where a prospect wants proof, we can turn round a bespoke example built specifically for their business before or during the pitch - a draft audit, a content calendar, a piece of the actual output an agent would produce for them. Turnaround and capability demonstrated on their own business, not described in the abstract.

09 · What's needed

The scaling question is answered. Here's what's left to build.

The obvious worry with a pitch like this is "does this cap out once you've signed a handful of clients." It doesn't, and the reason is infrastructure, not ambition. Every agent is software - a process that makes calls out to Anthropic's Claude API for the actual thinking, not something that needs a specific machine sitting in one place. That means client capacity runs on ordinary rented cloud servers, not on however many machines BGV happens to own, and it's the same software-isolation pattern already proven internally by Ted, our own agent, serving multiple people from one machine today. Here's what that unlocks, and what's still left to build around it.

The model: rent a general-purpose cloud server, host several clients' agents on it side by side in their own isolated workspace - separate credentials, separate memory, separate everything - and let each one call out to Anthropic's API as normal. No GPU, no specialist hardware, no dedicated machine per client. Capacity scales by renting another server when one fills up, a small and immediate step rather than a capital purchase, so the number of clients we can host is never the constraint on how many we can sign.

Infrastructure - solved

Hosting scales by renting servers, not buying hardware

Each client's agents run in their own isolated workspace on a shared, rented cloud server - the same multi-tenant pattern already proven internally. Adding the next client is a case of provisioning a workspace, not standing up new hardware, which means onboarding speed is set by our build process, not by procurement.

Isolation

Hard separation between clients' data

The same isolation rules that keep our own agents' memory and context private extend to client agents by design - separate directories, separate credentials, separate state per client on any shared server. No client's emails, credentials, or conversations are ever visible to another's, or to us by accident.

Access & credentials

Per-client integration setup

Every agent needs its own secure connection into the client's own email, calendar, CRM, and socials - not ours. That's a repeatable onboarding step Felix needs to template, not rebuild by hand for each new client.

Monitoring

Watchdog coverage across every client agent

We already catch and restart our own agents when they wedge or crash. That has to cover client-facing agents too, with someone actually on the hook to notice and fix it - a client's agent going dark for a day is a lost account, not an internal inconvenience.

Engineering capacity

A cap on how many builds Felix can take on

Felix is currently the only person who can build and configure a new agent end-to-end. Until that's documented into a repeatable build process someone else can run, the number of clients we can onboard is limited by his hours, not demand.

Cost tracking

Usage metering per client

API/compute cost scales with how hard each client's agent works. We need to actually track that per client so pricing is based on real cost, not a guess.
