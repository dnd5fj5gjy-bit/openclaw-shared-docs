Module 01

What Claude Code actually is.

Claude Code is Anthropic's command-line agent. The difference from a normal chat window: it doesn't just talk about your files, it reads them, edits them, runs commands, and checks its own work — in a loop, until the task is done. You give it a goal in plain English; it goes and does the work, the same way a capable junior colleague would, and comes back with a result rather than a suggestion.

This is exactly how Ted operates day to day: reading real files, drafting real documents, checking real state, rather than describing what someone else should do. That's the whole point of the tool — it closes the gap between "here's an idea" and "here's the finished thing."

Where it runs

On your machine (or a server), inside a terminal. It works directly on your actual project or folder — not a sandboxed toy version of it. That's what makes it powerful and also why the permission model (Module 05) matters.

Module 02

Getting started.

Three things to have in place before your first real session:

Step 1

Install and launch

Install via npm install -g @anthropic-ai/claude-code (needs Node.js). Then, from inside the folder you want to work in, type claude in the terminal. That folder becomes the agent's working directory — it can only see and touch what's inside it.

Step 2

Log in once

First run prompts you to authenticate with your Anthropic / Claude account (or an API key if you're on a team plan). This is a one-time step — it's remembered after that.

Step 3

Say what you want, in one message

No special syntax needed. Type the task as you'd brief a person: what you want, why, and any constraint that matters. Then let it work — don't narrate every step for it.

> Read the Q3 numbers in /reports and draft a one-page summary for the board, flag anything that looks off
Module 03

The core loop — how it actually works.

Every task follows the same pattern, and understanding it is what makes the tool predictable rather than mysterious.

1 · Understand

It reads before it acts.

It looks at the relevant files, checks existing context, and figures out what's actually there before touching anything.

2 · Act

It uses tools, not just words.

Reading files, writing files, running commands, searching the web — each is a discrete, visible action, not a black box.

3 · Check

It verifies its own output.

Good sessions include the agent re-reading what it produced, running a check, or confirming the result matches the ask.

4 · Report

It tells you what changed.

You get a plain-English summary at the end — not a wall of logs. If it's unsure or blocked, it asks rather than guessing.

Module 04

Project memory — CLAUDE.md.

The single highest-leverage habit: put a file called CLAUDE.md in the root of your project. It's loaded automatically at the start of every session — think of it as the standing brief you'd otherwise repeat every single time.

What belongs in it

  • Who you are and what this project is — one paragraph, not a biography.
  • Standing rules — things you'd otherwise correct every session ("never send emails without approval", "use hyphens not em dashes").
  • Where things live — folder structure, key files, naming conventions.
  • What "done" looks like — house style, formatting expectations, quality bar.

Everything Ted does for you already runs on exactly this pattern — it's why Ted doesn't re-ask the same questions every conversation.

Module 05

Permissions — how it stays safe.

Claude Code asks before doing anything it considers risky — editing a file for the first time, running a shell command, deleting something. You approve or deny each one. Three modes control how often it asks:

ModeWhat it meansBest for
DefaultAsks before every meaningfully risky action.First sessions, unfamiliar projects.
Auto-accept editsFree to edit files in the working folder without asking each time; still confirms bigger risks.Once you trust the task and want speed.
Plan modeReads and thinks, proposes a plan, does nothing until you approve it.Big or irreversible changes — see the plan before it acts.

The rule of thumb: reversible, local actions (reading, drafting, editing your own files) can run freely. Anything that leaves your machine or is hard to undo — sending something, pushing code live, deleting in bulk — should always get a confirmation, no matter how much you trust the tool.

Module 06

Using it well vs. fighting it.

Do
  • Brief it like a smart colleague — the goal, the constraint, the "why."
  • Give it the whole task in one message and let it run, rather than micromanaging each step.
  • Let it read the actual file/data before you describe what's in it — it's usually faster and more accurate than you summarising first.
  • Correct it once, clearly, and it'll hold the correction for the rest of the session.
Don't
  • Type one tiny instruction at a time — that's slower than one good brief.
  • Repeat context it can just read itself off disk.
  • Let a long session run without checking in — long stretches of unsupervised low-value work waste time either direction.
  • Approve destructive actions (delete, force-push, send) on autopilot — that's the one place manual review earns its keep.
Module 07

The power tools, once you're comfortable.

Slash commands

Reusable recipes

Type / to see built-ins like /init (auto-generate a starter CLAUDE.md) or /review. You can also define your own — a saved prompt you trigger by name instead of retyping it.

Subagents

Delegating side-work

For a big side-question ("go research X" / "find every place Y is used"), Claude Code can spin off a subagent that works in its own space and reports back a clean answer — so the side-quest doesn't clutter the main conversation.

Hooks

Automatic behaviour

Rules like "run the tests every time a file is saved" or "auto-publish anything written to /docs" — configured once, then they just happen. This is exactly how Ted's own auto-publish pipeline works.

MCP

Connecting outside tools

MCP servers plug in real integrations — Gmail, Calendar, Drive, a database — so the agent can act on those systems directly rather than you copy-pasting between windows.
