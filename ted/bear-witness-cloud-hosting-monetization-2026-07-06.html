Bear Witness · Cloud Hosting & Monetisation Model

Rent the compute, not the ceiling

The Mac Studio was never meant to be the product - it was the machine we built the agents on. This deck validates the alternative: renting cheap, multi-tenant cloud servers to host client agent instances, removing the capital constraint that would otherwise cap Bear Witness at however many machines BGV can afford to own.

See the model
01 · The core insight

The Mac Studio proved the agents work. It cannot be how they scale.

The pricing model already built for Bear Witness amortises one Mac Studio across the client book, and the maths in that deck is genuinely good - the machine costs £51/client at 3 clients and £7.70/client at 20. But that's amortisation, not scale. Every new client still runs on the same physical machine sitting in one location. Push past whatever that one machine can concurrently run and there are only two options: buy Olly or Felix a second Mac Studio per some client threshold, or ask a client to buy their own. Both reintroduce capital expenditure into a pitch whose entire premise is "low-cost AI agency" - and both cap growth at the rate BGV can afford to buy hardware, not at the rate it can win clients.

The fix is to stop treating "the agent" as something that lives on a specific physical machine at all. An agent, in the form Bear Witness actually runs it, is a Claude Code CLI process plus a set of scripts and memory files, making API calls to Anthropic's cloud for every actual thinking step. The process itself does no local inference and needs no GPU - it is a lightweight, mostly-idle piece of software waiting on network responses. That kind of process can run on any general-purpose Linux server, and several of them can run side by side on one modest, cheap, rentable box with no more isolation than separate directories and separate credentials per client. Decoupling the hosting layer from BGV-owned hardware turns infrastructure from a fixed, lumpy capital cost into a small, linear, rentable operating cost - which is what actually lets a team of three or four service dozens or hundreds of clients without a proportional increase in either headcount or capital.

This is not a hypothetical pattern. Ted - the very agent producing this deck - already runs exactly this shape of system today: a single machine serving multiple Telegram users concurrently, with per-user isolation handled entirely in software (separate memory directories, separate state files, separate credentials), not separate hardware. The multi-tenant-agent-hosting model proposed here is the same pattern, rented rather than owned, and offered to paying clients rather than used internally.

02 · How it actually works, technically

One rented Linux server, several isolated client agent workspaces, zero local model hosting.

The mechanism has four parts. First, rent a general-purpose cloud VM from a mainstream provider - a modest 2 vCPU / 4GB Linux instance is enough, since the process is network-bound, not compute-bound. Second, install the Claude Code CLI and Bear Witness's own agent scripts and memory-management tooling on that box. Third, create one isolated workspace directory per client on the same VM, each with its own credentials, memory files, and state - the same software-level separation Ted already uses across Telegram users. Fourth, and most importantly: every actual model call from every client's agent still goes out over the network to Anthropic's Claude API, exactly as it does today. The rented VM is doing orchestration and file/state management only - it never runs the model itself, so it never needs a GPU, and its own compute requirement stays small regardless of how "smart" the agent's output needs to be.

This deck is deliberately scoped to that hosting/orchestration layer only. Claude API token cost - the cost of the actual model calls - was already modelled per tier in the Bear Witness pricing deck (£42-456/month in API compute depending on tier, before this deck's hosting figures are added) and is not repeated or changed here. What follows is new: the cost of the box the agent process runs on, and how that cost behaves as more clients are added to it.

03 · Unit economics of multi-tenant hosting

Hosting cost per client falls as clients are packed onto the same box - up to a sensible ceiling.

A Claude Code CLI agent process is mostly idle: it wakes up on a heartbeat, a scheduled task, or an inbound message, does some lightweight file I/O and orchestration, sends the actual "thinking" work to Anthropic's API over the network, and goes quiet again waiting for the response. Independent reports of Claude Code's own process footprint put a single instance at roughly 270-370MB of RAM under normal use, with background/orphaned processes typically settling lower, around 60-170MB. CPU usage for an idle-most-of-the-time agent is negligible outside brief bursts around each turn.

Modelled assumption - not a sourced fact, flag to challenge
  • Per-client RAM footprint: modelled at 300-400MB per client agent workspace (idle CLI process + scripts + working memory files), toward the higher end of the observed 270-370MB range above, for headroom.
  • Usable RAM per box: on a 4GB VM, roughly 1-1.2GB is reserved for OS, the CLI runtime baseline, and safety margin, leaving ~2.8-3GB usable for client workspaces.
  • Conservative capacity ceiling: 8 client agent-workspaces per 4GB / 2 vCPU VM, run concurrently with headroom for occasional simultaneous bursts across clients. A less conservative packing (smaller footprint, more RAM headroom used) could plausibly push this to 10-12 - 8 is used throughout this deck as the deliberately cautious number.
  • Scale-up path: once a box approaches its 8-client ceiling, the next client is provisioned on a second VM at the same per-box price - cost scales in discrete steps (rent another box), not continuously, but each step is small (~£3-19/month) relative to a client's monthly retainer.

Because the per-box cost is flat and the client-per-box density is held constant, £/client hosting cost is close to flat rather than dramatically falling with scale - the real win here isn't a shrinking hosting line (it was already tiny), it's that hosting never becomes a bottleneck or a capital ask at any client count modelled. The Mac Studio's amortisation curve and this VM model's flat curve tell the same underlying story from two directions: shared infrastructure gets cheaper per client as the book grows, and even in the worst case here, infrastructure spend stays a rounding error against revenue at any realistic client count. Growth is gated by sales and onboarding capacity, not by servers.

04 · Growth model - small team, large company

Real, current examples of lean teams running large-scale, API-leveraged businesses.

Three current, citable examples of teams staying small while revenue scaled through cloud/API leverage rather than headcount: Midjourney generated roughly $200M in revenue in 2023 with 11 employees (~$18M revenue/employee that year), and as of 2026 runs at an estimated $500M ARR with a still-lean team of around 107 people (~$4.7M revenue/employee) - for comparison, Google runs at roughly $1.8M revenue/employee and Meta at roughly $1.6M. Gamma reached $50M ARR and 50 million users with a team of 30, remaining profitable for 15 consecutive months. Anysphere (maker of Cursor) took its Claude-powered coding agent from roughly $100M annualised revenue in January 2025 to an estimated $4B annualised run rate by June 2026 while headcount grew to around 300 - still a small fraction of the headcount a traditional company would need to service that much revenue, though notably no longer a "tiny team" by mid-2026.

Where the analogy holds: in every one of these cases, the marginal cost of serving one more customer is a slice of rented cloud compute plus metered API/model cost - not a new hire. That is exactly the mechanism this deck proposes for Bear Witness's hosting layer. Where it breaks down: Midjourney, Gamma and Cursor are self-serve products with automated signup, billing, and support - a user can start and scale their own usage with zero human involvement from the vendor. Bear Witness, at least in its first 12-18 months, is a bespoke consultancy: onboarding, oversight hours, and escalations are real, human, per-client costs, already costed into the pricing deck's oversight-time line (3-20 hours/month depending on tier). The infrastructure argument transfers cleanly; the "zero marginal cost of a new customer" argument does not transfer in full until Bear Witness builds its own onboarding automation - a reasonable future milestone, not a claim made here.

05 · Path to the data-centre vision

A bridge, not a competing plan - Phase 1 funds and proves Phase 2.

The long-term ambition - BGV's own data centre running a private, potentially open-source model stack - stays exactly where it was. Nothing in this deck argues against it. What it argues is that Bear Witness does not need to wait for that fundraise to start monetising the agent-hosting layer, because the software stack that runs on a rented cloud VM today is the identical software stack that will run on BGV's own racks later. Moving from Phase 1 to Phase 2 is a change of host, not a rebuild.

Phase 1 - now

  • Rented multi-tenant cloud VMs (Hetzner / DigitalOcean / Lightsail), opex-only, no capital outlay.
  • Claude Code CLI + Bear Witness agent scripts, unchanged from what already runs internally.
  • Anthropic's Claude API handles all model inference - no model hosting, no GPU procurement.
  • Scales in small, linear steps: rent another £3-19/month box when a box nears its client ceiling.
  • Generates real client revenue and real cost-to-serve data - the evidence a fundraise conversation needs.
Bridge target

Phase 2 - post-fundraise

  • BGV-owned data centre / private compute, funded by the traction Phase 1 generates.
  • Same Claude Code CLI + agent scripts layer, migrated onto owned infrastructure with no re-architecture.
  • Optionality to run open-source / self-hosted models alongside or instead of the Claude API for specific workloads, once owned GPU capacity exists.
  • Full data sovereignty for clients who require it - no third-party cloud provider or model vendor in the chain for those workloads.
  • Better long-run unit economics at large scale, once utilisation is high enough to beat rented-cloud rates.

Phase 1 is the fundraising case, not a placeholder to be discarded. A working multi-tenant hosting layer with real paying clients on it is direct proof that the demand, the unit economics, and the operating model all work - exactly the evidence an investor or lender would want before funding a data centre. Skipping straight to Phase 2 without this would mean asking for capital on the strength of a pitch deck alone.

06 · Risks and honest caveats

What has to be managed, not glossed over.

Vendor dependency

Two points of external dependency, not one

This model depends on both Anthropic's Claude API remaining available and priced as expected, and on the chosen cloud provider's uptime and pricing. Neither is new risk introduced by this proposal specifically - the existing pricing deck already depends on Claude API pricing - but multi-tenant hosting adds a second vendor relationship (the cloud provider) that didn't exist when everything ran on BGV's own Mac Studio.

Multi-tenant isolation

Client data must not leak across tenants sharing a box

Software-level isolation (separate directories, separate credentials, separate process spaces per client) is the model proposed here, and it is what Ted already runs in production internally. For clients with stricter data-handling requirements, a safer but more expensive alternative exists: one VM per client, or containerisation (Docker) per client on a shared box. Rough cost delta: dedicated-VM-per-client at Hetzner CX22 pricing runs ~£3.26/client/month instead of ~£0.41/client/month shared 8-ways - roughly 8x the hosting cost, still a trivial absolute number against a £950-8,000/month retainer, and worth offering as a premium/compliance-tier option rather than the default.

Reliability ownership

Uptime is now BGV's problem, not the client's hardware

When agents ran on a client's own machine, hardware failure was the client's problem. Renting and operating the hosting layer moves uptime responsibility onto BGV - a box going down affects every client on it, not just one. This is manageable (mainstream providers publish SLAs, and boxes are cheap enough to run redundant pairs) but it is a genuine new operational responsibility that didn't exist under the old model, and should be reflected in how the service is scoped and supported.

Usage spikes - the actual strength

Elasticity cuts the right way here

Unlike the Mac Studio, which is a fixed physical ceiling, rented cloud compute can scale up or down within minutes - add a box when a client's usage spikes, resize an existing box, or shift a client to a dedicated instance if their volume outgrows shared hosting. This is the one place where "rented" beats "owned" outright: capacity elasticity that owned hardware structurally cannot match without buying ahead of demand.

07 · Recommendation and next steps
